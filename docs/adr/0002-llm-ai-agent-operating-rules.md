# ADR-0002: Правила работы LLM/AI-агентов с проектом

- Status: Accepted
- Date: 2026-04-26

## Context

Проект явно создавался и развивается с участием AI-агентов. Это ускоряет
разработку, но повышает риск случайных архитектурных разворотов, потери
неявных доменных инвариантов, правок чужих изменений и поверхностной проверки.

Нужны правила, которые будущие LLM/AI-агенты должны соблюдать перед изменениями,
во время реализации и при финальной проверке.

## Decision

Будущие LLM/AI-агенты должны считать этот ADR обязательным рабочим соглашением.

### Перед изменениями

1. Проверить `git status --short` и не перетирать изменения, которые агент не
   делал.
2. Прочитать релевантные ADR в `docs/adr`, `README.md`, `package.json`,
   `prisma/schema.prisma` и только затем переходить к целевым файлам.
3. Сначала найти существующий паттерн через `rg`/чтение соседних модулей.
   Новые абстракции добавлять только если они реально уменьшают сложность или
   совпадают с текущей структурой.
4. Если задача меняет архитектурное соглашение из ADR-0001, добавить новый ADR
   или обновить статус существующего ADR.

### Границы кода

1. Не смешивать HTTP-логику, бизнес-логику и UI:
   - HTTP остается в `src/app/api/**/route.js`.
   - Доменные правила остаются в `src/features/*/server.js`.
   - Клиентские fetch-обертки остаются в `src/shared/api`.
   - Общий UI и hooks остаются в `src/shared/ui`.
2. Не добавлять TypeScript, Tailwind, новые state managers, новые charting/UI
   библиотеки или auth framework без отдельного решения.
3. Не переносить бизнес-инварианты только в клиент. Клиент может быть
   оптимистичным, но сервер обязан валидировать доступ, суммы и связи.
4. Не менять public shape API payloads без обновления всех клиентов и явного
   описания миграции.

### Доменные инварианты

1. Деньги хранятся и передаются как целые minor units.
2. Период бюджета задается парой `year` и `month`.
3. Доступ к бюджету другого пользователя всегда проходит через `ownerId` и
   `ensureAccess(ownerId)`.
4. После мутаций бюджета нужно сохранять месячную статистику и пересчитывать
   накопления, если операция влияет на соответствующие агрегаты.
5. Категории, расходы, накопления и share-записи должны уважать cascade delete
   и связи Prisma-схемы.

### База данных

1. Изменения модели начинаются с `prisma/schema.prisma`.
2. Миграции должны быть добавлены в `prisma/migrations`, а не имитироваться
   ad hoc SQL в коде приложения.
3. Нельзя менять уже примененные миграции задним числом без явного запроса.
4. При изменении денежных полей, уникальных индексов, cascade rules или
   ownership-модели нужен ADR или обновление существующего ADR.

### Безопасность

1. Не логировать email/password payloads, session tokens, cookies и connection
   strings.
2. Не читать и не раскрывать `.env`, если для задачи достаточно `.env.example`.
3. Новые API-роуты должны явно проверять авторизацию.
4. Любое усиление auth должно быть совместимо с Redis-сессиями или сопровождаться
   ADR о замене сессионной модели.

### UI и Telegram

1. Сохранять поддержку обычного браузера и Telegram Mini App.
2. Не ломать CSS Modules/variables подход. Глобальные переменные менять
   аккуратно, потому что они влияют на весь UI kit.
3. Оптимистичные обновления Zustand допустимы, но серверный snapshot остается
   источником истины после ответа API.
4. При изменении модальных окон или auth flow проверять обработку Telegram
   MainButton и fallback-кнопок обычного браузера.

### Проверка

Минимальная проверка зависит от типа изменения:

- Документация: проверить diff и ссылки внутри docs.
- UI-only: запустить сборку или локально проверить страницу, если окружение
  доступно.
- API/business logic: добавить или выполнить targeted проверку route/domain
  сценария; если тестов нет, минимум проверить сборку и описать ручной сценарий.
- Prisma/schema: выполнить `npm run prisma:generate` и применимую Prisma
  проверку/миграцию в доступном окружении.

Если команда не запускается из-за отсутствующей БД, Redis или сетевого доступа,
агент должен явно написать это в финальном ответе и не выдавать изменение за
полностью проверенное.

## Consequences

- Последующие агенты получают явный контракт и меньше поводов делать
  несовместимые рефакторинги.
- Документационные изменения могут быть проверены без запуска приложения, но
  кодовые изменения должны сопровождаться релевантной проверкой.
- Архитектурные изменения станут чуть медленнее, потому что требуют ADR, зато
  проект сохранит накопленные решения и доменные инварианты.
