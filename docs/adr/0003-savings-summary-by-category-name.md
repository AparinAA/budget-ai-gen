# ADR-0003: Агрегировать накопления по названию категории

- Status: Accepted
- Date: 2026-04-26

## Context

Накопления хранятся в `SavingsTransfer` по `categoryId`, `year` и `month`.
Категории создаются отдельно для каждого месячного бюджета, поэтому одно и то
же пользовательское название категории в разных месяцах получает разные
`categoryId`. Из-за этого блок "Накопления" показывал несколько строк для
одинаковых названий вместо общей суммы по категории.

## Decision

`GET /api/budget/savings` должен отдавать серверный агрегат `categoryTotals`,
где накопления сгруппированы по отображаемому названию категории. Детальные
`transfers` остаются в ответе для обратной совместимости и возможных будущих
детализаций.

Ответ также может содержать:

- `totalCurrencies` - общий итог накоплений по валютам для корректного
  отображения общей суммы без смешивания minor units разных валют;
- `monthlyTotals` - общий итог накоплений по каждому месяцу с тем же
  валютным разбиением.

Каждый элемент `categoryTotals` содержит:

- `name` - отображаемое название категории;
- `amount` - общая сумма в minor units для совместимости с текущей моделью;
- `currencyCode` - код валюты, если все записи группы в одной валюте;
- `currencies` - суммы группы по валютам, чтобы не смешивать разные валюты в
  одном отображаемом значении;
- `transfersCount` - количество записей `SavingsTransfer`, вошедших в группу;
- `months` - помесячная детализация этой категории, где каждый элемент
  содержит `year`, `month`, `amount`, `currencyCode`, `currencies` и
  `transfersCount`.

Клиент блока "Накопления" должен использовать `categoryTotals`, а если поле
отсутствует или не содержит `months`, собрать такой же агрегат из `transfers`
как fallback.

## Consequences

- Блок "Накопления" отображает одну строку на название категории, например
  "Путешествия: общая сумма накопления", даже если накопления пришли из разных
  месячных категорий.
- По клику на агрегированную категорию клиент раскрывает помесячную
  детализацию накоплений для этой категории.
- API остается обратно совместимым: существующие клиенты, читающие `transfers`,
  не ломаются.
- Новые изменения в summary накоплений должны сохранять серверную агрегацию и
  фронтенд-fallback, если только публичный контракт API не будет пересмотрен
  отдельным ADR.
