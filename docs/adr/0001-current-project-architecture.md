# ADR-0001: Зафиксировать текущую архитектуру Family Budget

- Status: Accepted
- Date: 2026-04-26

## Context

Family Budget - небольшое веб-приложение для ведения семейного бюджета,
созданное в основном AI-агентом. В проекте уже сложилась рабочая архитектура:

- Next.js 14 App Router, React 18, JavaScript без TypeScript.
- CSS Modules и глобальные CSS-переменные без Tailwind.
- PostgreSQL как основное хранилище, Prisma как ORM и миграционный слой.
- Redis для серверных сессий и кэша курсов валют.
- Zustand для клиентского состояния текущего периода, снапшота бюджета,
  валюты и выбранного владельца бюджета.
- Recharts для графиков.
- Telegram Mini App интеграция на уровне layout, auth page и модальных окон.

Код распределен по слоям:

- `src/app` - страницы, layout, middleware и Route Handlers.
- `src/features` - серверная бизнес-логика доменов `auth` и `budget`.
- `src/entities` - доменные UI-компоненты бюджета.
- `src/shared` - общие инфраструктурные модули, API-клиенты, UI kit,
  форматирование и store.
- `prisma` - схема БД и миграции.

Доменная модель хранит деньги целыми числами в minor units. Бюджет уникален
для пары `userId + month + year`. Категории принадлежат бюджету, расходы
принадлежат категории и бюджету, статистика месяца и накопления денормализованы
отдельными таблицами.

Аутентификация реализована кастомно: email/password, серверные сессии в Redis,
HTTP-only cookie `session`, middleware для страниц `/`, `/budgets`, `/auth` и
проверки авторизации внутри API-роутов. Совместный доступ реализован через
`BudgetShare`, где `ownerId` обозначает владельца бюджета, к которому получает
доступ другой пользователь.

## Decision

Фиксируем текущую архитектуру как baseline для дальнейшей разработки:

1. Оставляем Next.js App Router и существующее JavaScript-окружение. Переход на
   TypeScript, Tailwind, другую ORM, другой роутинг или отдельный backend
   требует нового ADR.
2. Сохраняем текущие границы слоев:
   - Route Handlers в `src/app/api/**/route.js` отвечают за HTTP, парсинг
     запроса, статус-коды и JSON-ответы.
   - Бизнес-правила бюджета и авторизации живут в `src/features/*/server.js`.
   - Клиентские API-обертки живут в `src/shared/api`.
   - Переиспользуемые UI-компоненты и инфраструктура живут в `src/shared`.
   - Бюджетные виджеты верхнего уровня остаются в `src/entities/budget/ui`.
3. Все суммы в БД, API payload и доменной логике хранятся в minor units
   целыми числами. Конвертация в major units допускается только на границе UI и
   форматирования.
4. PostgreSQL/Prisma остается source of truth. Изменения схемы выполняются через
   `prisma/schema.prisma` и миграции в `prisma/migrations`.
5. Redis считается обязательной runtime-зависимостью для сессий. Код, который
   касается авторизации, должен учитывать `USER_CASH_REDIS_URL`.
6. Для бюджета сохраняется action-based mutation API через `POST /api/budget`
   до отдельного решения о переходе на granular REST endpoints или Server
   Actions.
7. Правила доступа к чужому бюджету централизуются через `ensureAccess(ownerId)`.
   Новые операции, читающие или меняющие бюджетные данные, должны проходить
   через эту проверку или явный эквивалент с тем же смыслом.
8. Telegram Mini App остается поддерживаемой средой. Изменения auth, layout,
   modal UX и кнопок действий не должны ломать обычный браузерный режим и
   Telegram WebApp режим.

## Consequences

- Новые фичи должны сначала искать подходящий слой в существующей структуре, а
  не создавать параллельные архитектурные ветки.
- Мутации бюджета должны обновлять месячную статистику и накопления, если они
  меняют доход, категории, расходы, валюту или rollover-логику.
- Изменения схемы БД должны сохранять инварианты `userId + month + year` для
  бюджета и целочисленное хранение денег.
- Появление полноценной системы ролей, viewer-only доступа, stronger password
  hashing, refresh/rotation sessions или отказа от Redis должно оформляться
  отдельным ADR.
- Тестов и реального lint сейчас нет: `npm run lint` является заглушкой.
  Значимые изменения должны добавлять точечную проверку там, где это снижает
  риск, либо явно фиксировать, что проверка ограничена сборкой/ручным сценарием.

## Known Risks

- `passwordHash` сейчас строится через SHA-256 без соли и специализированного
  password hashing алгоритма. Это технический долг безопасности.
- `.env.example` документирует только `DATABASE_URL`, но runtime-код также
  использует `USER_CASH_REDIS_URL`.
- `BudgetShare.role` уже хранится, но текущая бизнес-логика фактически
  трактует доступ как editor-доступ.
- API `shareBudget(budgetId, ...)` на клиенте по смыслу передает ownerId, а не
  id конкретного бюджета. Это важно учитывать при рефакторинге.
